# CMF 与其他模型格式的对比

本文对 **CMF**（Cortiq Model Format，参见 [CMF_V2_SPEC.md](CMF_V2_SPEC.md)）
与最常被拿来比较的模型容器和引擎格式做一次诚实、务实的对比：

- **GGUF** —— `llama.cpp`/`ggml` 使用的单文件容器。
- **safetensors** —— Hugging Face 的零拷贝张量容器。
- **ONNX** —— 跨框架的计算图 + 权重交换格式。
- **PyTorch `.pt` / `.pth`** —— 经 pickle 序列化的状态字典 / 检查点。
- **GGML（旧版）** —— GGUF 之前的 `ggml` 文件格式。
- **TensorRT 引擎** —— NVIDIA 序列化的、与特定硬件绑定的计划（plan）文件。

一个需要反复强调的前提：这些并不都属于同一*类*东西。GGUF、
safetensors、ONNX、`.pt` 和 GGML 主要是**存储/交换格式**；
TensorRT 引擎是一种**编译后的运行时产物**；而 CMF 既是一种存储
格式，又附带一个参考 CPU 运行时。下表按人们真正关心的维度进行
对比，但请结合正文来理解每种格式实际所面向的形态。

## 特性对比表

图例：**是** = 格式本身支持；**部分** = 可行但有条件、需外部支持
或不完整；**否** = 格式未提供；**不适用** = 对该类产物不适用。

| 单一自描述文件 | 是 | 是 | 部分 | 部分 | 部分 | 部分 | 是 |
| 可内存映射 / 零拷贝加载 | 是 | 是 | 是 | 部分 | 部分 | 是 | 否 |
| 量化内建于格式中 | 是 | 是 | 否 | 部分 | 部分 | 是 | 是 |
| 内嵌分词器 | 是 | 是 | 否 | 否 | 否 | 部分 | 否 |
| 内嵌对话模板 | 是 | 是 | 否 | 否 | 否 | 否 | 否 |

### 对具体单元格的说明（以便这些标记站得住脚）

- **CMF —— 自描述 / 完整性：** 固定的 128 字节信封、二进制张量
  目录，以及每张量 + 每分节的 `hash64` 都是强制性的；`cortiq verify`
  会校验整条链。分词器和对话模板*在存在时被支持并内嵌*（可选
  分节），而非强制写入每个文件。
- **CMF —— 流式 = 部分：** CMF 支持惰性 `mmap` 分页（冷权重不占用
  RSS）和多文件分片，但它并不是一种网络流式 / 渐进下载协议。我们
  诚实地把它标为“部分”，而非“是”。
- **CMF —— 技能叠加：** 这是 CMF 真正与众不同的地方 —— 多个技能
  通过全形状替换张量共享同一个基座，并从单一目录寻址（参见规范
  §9）。GGUF 的“部分”指的是在加载时应用的*外部* LoRA 适配器；
  TensorRT-LLM 的“部分”指的是运行时的多 LoRA，而不是引擎文件本身
  的属性。
- **GGUF / GGML —— 完整性 = 否：** 这些容器带有魔数 + 版本号，但
  没有内建的逐张量内容哈希。损坏无法被格式本身检测到。
- **safetensors —— 运行时 = 不适用：** 它是一种带有小型加载库的
  存储格式，而非推理运行时；“纯 CPU、无依赖的运行时”不适用于它。
- **ONNX / PyTorch —— 单一文件 = 部分：** 两者都常把大权重溢出到
  外部数据文件（ONNX 的外部数据；分片 / `zip` 检查点），因此“单个
  自包含文件”并无保证。

## 逐格式详解

### GGUF（llama.cpp）
在设计意图上，GGUF 是与 CMF 最接近的同类：单个自描述文件，包含
权重、丰富的键/值元数据块、内嵌分词器和对话模板，以及一大批内建
量化方案（k-quants 和 i-quants）。它久经沙场，对 `mmap` 友好，背后
有庞大的 `llama.cpp` CPU/GPU 生态。相对于 CMF，它的主要缺口在于缺少
内建内容哈希，以及缺少一等公民式的“单基座 / 多技能”叠加（LoRA
是作为独立适配器应用的）。当你想要最大的工具链/硬件覆盖面，以及
一套当下已被验证的本地推理栈时，选择 GGUF。

### safetensors
safetensors 把一个问题解决得极为出色：安全、快速、零拷贝的张量
存储，且没有任意代码执行风险（不像 pickle）。它是 Hugging Face
生态中分发原始权重的事实标准。它有意*只*存储张量加上一小段 JSON
元数据头 —— 不含架构、分词器、对话模板或量化方案 —— 因此一个模型
实际上是一个文件目录（`config.json`、`tokenizer.json`，以及一个或
多个 `.safetensors`）。当你想要一个安全、与框架无关的权重容器，且
乐于把配置和分词器作为附带（sidecar）文件携带时，选择 safetensors。

### ONNX
ONNX 是一种“计算图加权重”的交换格式：它捕获的是计算过程，而不
只是参数，这让它在跨框架可移植性上很有优势，也便于通过 ONNX
Runtime 在众多运行时和加速器上运行同一个模型。它支持量化算子
（QDQ / int8），但没有内嵌分词器或对话模板，未针对 `mmap` 零拷贝
设计，且大模型会把权重拆分到外部数据文件中。当你的首要目标是跨
框架和跨部署目标的互操作性，或者当你需要完整的计算图而非固定的
LLM 架构时，选择 ONNX。

### PyTorch `.pt` / `.pth`
这些是经 pickle 序列化的 Python 对象 —— 通常是一个 `state_dict`，
有时是整个模型或训练检查点。它们是 PyTorch 内部原生、最灵活的
格式，在训练和研究阶段非常理想。但用于分发的缺点也确实存在：
pickle 在加载时可能执行任意代码（一个真实的安全隐患），没有模型
类就无法自描述，没有内嵌分词器/模板，而且加载需要完整安装
PyTorch。在训练、保存检查点和 PyTorch 内部工作流中选择
`.pt`/`.pth`；对外发布时，请转换为更安全的容器。

### GGML（旧版）
GGML 是 `llama.cpp` 背后最初的单文件格式，后被 GGUF 取代。它提供
了 `mmap`、内建量化和内嵌词表，并验证了“一个小文件、CPU 推理”
这一理念。但它的元数据有限，且常常预设了架构，没有可扩展的键/值
块，没有对话模板，也没有完整性哈希 —— 而这些正是 GGUF 取代它的
原因。这里列出它仅供参照；新工作应面向 GGUF（或 CMF），而非
GGML。

### TensorRT 引擎
TensorRT 引擎（`.plan` / `.engine`）不是一个可移植的模型文件 ——
它是针对特定 GPU、TensorRT 版本，往往还针对特定批量/形状剖面
编译并优化后的产物，量化（INT8/FP8）在构建期就已烘焙进去。在
这些约束之内，它能提供顶级的 NVIDIA GPU 吞吐与延迟，并在反序列化
时防范版本/硬件不匹配。它不透明、不可移植、不面向 `mmap`/CPU，
也不携带分词器或对话模板。当你要部署在 NVIDIA GPU 上、追求极致
推理性能，并且能够为每个目标环境重新构建时，选择 TensorRT 引擎。

## 何时选择 CMF
当你想要一个单一、自描述、经完整性校验的文件，用于**以 CPU 为先、
内存映射**的推理，并让分词器和对话模板随模型一同携带时，选择
CMF；而其最具区分度的特性在于 —— 当你希望**多个面向不同任务的
专用技能共享同一个基座**、同处一个文件（或一组分片）之中，以全
形状替换张量的方式叠加，而无需为每个技能各自组装一个独立模型时，
选择 CMF。它内建的双字段与可变比特量化、强制性的逐张量与逐分节
哈希，以及无依赖的参考运行时（Rust，外加一个仅需标准库加 numpy 的
Python 读取器），使它非常适合在通用硬件上分发和服务紧凑、可路由、
可验证的模型。反之，如果你需要当下最广泛的现成工具链，请选 GGUF；
如果你需要一个面向 HF 生态的、纯粹而安全的权重容器，请选
safetensors；如果你需要跨框架的计算图可移植性，请选 ONNX；如果你
需要 NVIDIA GPU 的极致延迟，请编译一个 TensorRT 引擎。CMF 有意
比通用图格式更聚焦，而且与 GGUF 不同，它较新、生态更小 —— 这正是
为换取其技能叠加与完整性保障所付出的、坦诚的代价。
