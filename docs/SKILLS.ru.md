# Навыки: рой специалистов в одном файле

*Языки: [English](SKILLS.md) · [中文](SKILLS.zh.md)*

Один `.cmf`-файл несёт **базовую модель** плюс любое число **навыков** —
наборов замещающих тензоров. Рантайм читает тензоры навыка *вместо*
тензоров базы (никогда не складывает и не собирает вторую модель),
поэтому переключение специалистов бесплатно: тот же mmap, та же память,
другие веса по тем же адресам. Промпт направляется к нужному навыку
автоматически — файл сам знает, какой из его специалистов подходит, без
обученного гейта и внешнего роутера.

## Термины — три слова, которые нужно знать

- **База (базовая модель)** — обычная LLM, сконвертированная в CMF
  (`cortiq convert`). Это общие веса модели *без* навыков; навыки лежат
  в том же файле рядом и подменяют часть её тензоров, когда активны.
- **Навык** — набор тензоров-замен («в этих слоях FFN возьми веса
  специалиста, остальное — у базы») плюс дескриптор роутинга ~4 КБ,
  по которому файл сам узнаёт «этот промпт — мой».
- **Донор** — *дообученный чекпойнт той же самой базы* (SFT или
  влитая LoRA), из которого навык можно взять готовым. Донор — это НЕ
  другая модель («кодер», «философ»): прививка из родственника после
  continued-pretrain не работает, и ниже это измерено.

## Откуда берётся навык: два пути

**Путь 1 — датасет, без донора: `cortiq skill bake`.** Если у вас есть
только база и тексты домена (корпус, задачи бенчмарка), движок сам
находит нужные нейроны. Это ровно интуиция «подсветки» из imatrix,
доведённая до конца: прогоняем корпус через модель, смотрим, какие
FFN-нейроны несут задачу, — но вместо суммирования «по ИЛИ» маска
*обучается* (L1-регуляризация по DTG-MA — заявка USPTO 19/452,464,
[PATENTS.md](../PATENTS.md)), потому что измеренная
честно top-K-подсветка без обучения рушит качество (PPL 59 против 11.9 —
см. ниже). Обученная маска отбрасывает шумовые нейроны — и модель на
домене сначала становится *лучше*, потом последние слои полируются
против точного учителя (FCD). Никакой второй модели не нужно.

**Путь 2 — готовый донор: `cortiq skill add`.** Если кто-то уже дообучил
вашу базу (text-to-SQL-тюн, русский SFT), обучение не нужно вообще:
«засветка» уже сделана автором донора — его файнтюн и есть результат
обучения. Остаётся выделить изменённое: каждый тензор донора сравнивается
с базой, нетронутые тюном выбрасываются (`--min-delta`), изменённые
прививаются в файл как навык. Минуты на ноутбуке, никакого GPU.

Оба пути заканчиваются одинаково: в файле появляется навык с id, именем
и дескриптором роутинга, `skill list` его показывает, `--quality` пишет
измеренный вердикт в реестр.

### FAQ

**База — это веса вместе с масками/скиллами?** Нет. База — общие веса
модели. Навыки и маски — отдельные записи каталога в том же файле; при
активации навыка рантайм читает его тензоры вместо базовых.

**Как выделить нужные веса из донора без обучения?** Обучение уже
сделано — донором. Мы не ищем «нейроны навыка» в доноре; мы берём
*разницу*: тензоры, которые файнтюн изменил (референс-декодер меряет
относительную дельту к базе), и только их храним. Порог — `--min-delta`.

**А если донора нет?** Тогда путь 1: датасет → `skill bake` → обученная
маска + полировка. Это и есть сценарий «прогнал 50 задач бенчмарка —
получил swe-skill»: `--files` принимает ваши тексты, всё остальное
движок делает сам (реальный лог прогона — ниже).

**Можно передавать навыки друг другу?** Да. Навык — это записи каталога
внутри `.cmf`; файл со своим роем можно отдать целиком, а перенос
отдельного навыка между файлами одной базы — операция над каталогом
(без пересчёта весов). Skill-hub — ровно то направление, куда это ведёт.

Всё ниже воспроизводится тремя командами на настоящих чекпойнтах с
Hugging Face; файлы промптов и оценок лежат в
[`docs/skills-demo/`](skills-demo/). Замерено на `cortiq` 0.3.4+ с
Qwen2.5-0.5B-Instruct (q8) в роли базы.

## Демо: один файл, четыре личности

```sh
# 1. База
cortiq convert --model Qwen/Qwen2.5-0.5B-Instruct --quant q8 --output swarm.cmf

# 2. Три навыка из настоящих дообучений той же базы
cortiq skill add swarm.cmf --from vindows/qwen2.5-0.5b-text-to-sql-merged \
  --id sql --name "SQL assistant" --prompts docs/skills-demo/prompts-sql.txt

cortiq skill add swarm.cmf --from Vikhrmodels/Vikhr-Qwen-2.5-0.5b-Instruct \
  --id ru --name "Русский ассистент" --prompts docs/skills-demo/prompts-ru.txt \
  --quality docs/skills-demo/eval-ru.txt

cortiq skill add swarm.cmf --from ewre324/ewre324-Thinker-Qwen2.5-0.5B-Instruct-Reasoning \
  --id thinker --name "Step-by-step verifier" --prompts docs/skills-demo/prompts-think.txt
```

`skill list` показывает, что теперь несёт файл:

```
3 skill(s):
  sql        SQL assistant            72 tensor(s), 314.3 MB, layers [0..23], routable
  ru         Русский ассистент        72 tensor(s), 314.3 MB, layers [0..23], routable
      quality: {"backbone":11.875,"overlaid":11.027,"metric":"ppl","file":"eval-ru.txt"}
  thinker    Step-by-step verifier    72 tensor(s), 314.3 MB, layers [0..23], routable
```

Один файл 1.45 ГБ вместо четырёх отдельных моделей по 479 МБ (1.9 ГБ),
один mmap, а переключение личностей ничего не стоит при загрузке — тензоры
навыка это обычные записи каталога, на которые рантайм указывает вместо
тензоров базы.

## Что измеримо стало лучше

**Роутинг: 6/6 на промптах, которых роутер не видел.** Файл направляет
свежие промпты к нужному специалисту с 3–5× запасом по ошибке
(`E = ‖r−BBᵀr‖²/‖φ‖²`, меньше — ближе):

```
$ cortiq route swarm.cmf -p "Show me the SQL to find orders with no matching invoice."
  sql      E = 0.0016   ← победитель
  thinker  E = 0.0073
  ru       E = 0.0112

$ cortiq route swarm.cmf -p "Расскажи, как приготовить сырники из творога."
  ru       E = 0.0050   ← победитель
  thinker  E = 0.0168
  sql      E = 0.0204

$ cortiq route swarm.cmf -p "Verify step by step whether 91 is divisible by 7 and by 13."
  thinker  E = 0.0047   ← победитель
  sql      E = 0.0165
  ru       E = 0.0190
```

**`ru`: −7.1% перплексии на русской прозе**, записано в реестр самого файла
при запекании (база 11.88 → с навыком 11.03 на отложенном русском
тексте). Разница видна и в генерации: база выдаёт кольцевые пункты
(«Стоимость аренды обычно выше, чем лизинг… Стоимость лизинга обычно
ниже»), навык пишет связную структурированную прозу.

**`thinker`: проверяет, а не утверждает.** Спросите *«Is 17077 a prime
number?»* — база уверенно выдумывает ложную факторизацию
(`17077 = 3 × 5699`, что неверно: 17077 — простое); навык начинает
систематическую проверку делимости. Тот же файл, `--skill thinker`.

**`sql`: структура text-to-SQL.** Навык отвечает на вопросы по схеме
оконными функциями и CTE там, где база пишет плоские джойны.

Честно о метриках: перплексия — правильный гейт для навыка, сдвигающего
*распределение* домена (как `ru` на русском тексте). Для
инструкционных SFT-доноров (`sql`, `thinker`) выигрыш — в *поведении
генерации*, а не в правдоподобии текста: меряйте их A/B-генерациями или
точностью на задачах и не удивляйтесь, что PPL на рукописном корпусе
стоит на месте или чуть хуже. `--quality` записывает в реестр то, что вы
измерили; он никогда молча не «освящает» навык.

## Как пользоваться

```sh
cortiq run swarm.cmf -p "..."                      # роутинг сам: файл выбирает специалиста
cortiq run swarm.cmf -p "..." --skill thinker      # закрепить явно
cortiq run swarm.cmf -p "..." --skill none         # принудительно базовая модель
cortiq route swarm.cmf -p "..."                    # оценить все навыки
cortiq explain swarm.cmf --prompt "..."            # роутинг + распределение первого токена
cortiq run swarm.cmf -p "..." --blend auto         # мягкая суперпозиция топ-2
cortiq run swarm.cmf -p "..." --route-dynamic --trace   # переключение по токенам
```

`--blend auto` смешивает топ-2 навыка с весами softmax(−E/T) — промпт с
проверкой арифметики на хорватском лёг между `ru` и `thinker` примерно
50/50. `--route-dynamic` переключает активный навык *посреди потока* по
мере смены контекста, решение на каждом токене принимает та же ошибка
реконструкции E по φ-EMA. Реальный трейс (`--trace`, смешанный промпт
«Найди в базе всех клиентов без заказов: напиши SQL-запрос и объясни
его по-русски», `CMF_ROUTE_EON=0.9 CMF_ROUTE_EOFF=1.0`):

```
  #  токен        навык       E
  0  Вот          —           —       первые токены: роутер копит φ-EMA
  7  ,            —           0.857   E < EON(0.9) → включение
  8  который      thinker     0.857   активен thinker
 23  sql          thinker     0.662   дно E — генерация SQL-блока
 31  c            thinker     0.834   русское объяснение: E растёт…
 47  ет           thinker     0.865   …но E < EOFF(1.0) — гистерезис держит
```

Числа читаются как запас: между «включиться» (0.857 против порога 0.9)
и «выпасть» (0.865 против 1.0) остаётся зазор ~0.13 — дребезга нет.
Пороги по умолчанию (EON 0.62 / EOFF 0.74) консервативны: на этом рое
дно E — 0.662, то есть без настройки файл честно остаётся на базе.
Поднимайте `CMF_ROUTE_EON` / `CMF_ROUTE_EOFF`, когда φ-статистики ваших
навыков лежат дальше от промптов, чем в этом демо.

## Путь 2 подробно: `cortiq skill add` — привить донора

`cortiq skill add` прививает тензоры **донора** в контейнер базы:

```sh
cortiq skill add <base.cmf> \
  --from <hf-репозиторий-или-локальная-папка>  # донор той же архитектуры
  --id <id> [--name "..."]        # идентичность в реестре
  --layers all|A-B|i,j,k          # какие слои (по умолчанию все)
  --tensors ffn|attn|all          # какие семейства (по умолчанию ffn)
  --prompts file.txt              # 8+ примеров промптов → подпространство роутинга
  --quality held-out.txt          # PPL-гейт, записывается в реестр
  --min-delta 0.02                # выбросить тензоры, которых тюн не касался
  --skill-quant vbit --mean-bits 6  # более дешёвая кодировка оверлея
  --output out.cmf                # по умолчанию: перезапись на месте
```

Тензоры донора переквантуются кодировкой самой базы (q8-база получает
q8-навыки); прививка битово честна: перезапекание навыка из
*собственного* исходного чекпойнта базы меняет PPL ровно на +0.0%.

**Берите доноров-дообучения вашей же базы.** Это единственное
правило, решающее исход. FFN-тензоры со-адаптированы с вниманием вокруг:

- **SFT / merged-LoRA той же базы — работает.** Все три донора демо —
  дообучения Qwen2.5-0.5B-Instruct; их FFN-прививки когерентны и
  поведенчески различимы.
- **Родственники после continued-pretrain — как частичная прививка не
  работают.** Мы измерили, чтобы вам не пришлось: прививка FFN (и даже
  FFN+attention) из Qwen2.5-**Coder**-0.5B-Instruct — те же формы, та же
  семья, но глубоко дообученный на код сиблинг — даёт PPL 238 против 2.6
  у базы. Его веса со-адаптированы с нормами и эмбеддингами, которых
  навык не несёт. Если донор уехал так далеко — нужен отдельный файл
  модели, а не навык.

**Промпты роутинга** (`--prompts`): 8 коротких промптов, которые *звучат
как пользователи навыка*. По ним фитятся статистики φ(x) (среднее +
PCA-базис ранга 2 над усреднённым скрытым состоянием на 2/3 глубины) и
сохраняются в заголовок — ~4 КБ на навык, дальше файл маршрутизирует себя
сам. Навыки, замещающие не-FFN тензоры, исключаются из *динамического*
роутинга (статический `--skill` работает) — рантайм предупреждает об этом.

## Путь 1 подробно: `cortiq skill bake` — из датасета, без Python

Весь рецепт DTG-MA выполняется нативно — обучение маски, FCD-полировка
и defrag-запекание одной командой на CPU (тренировочные GEMM едут тем
же Accelerate-путём, что и префилл; attention заморожен, поэтому
backward идёт только по FFN-цепи):

```sh
cortiq skill bake base.cmf \
  --files docs/CMF_V2_SPEC.ru.md README.ru.md docs/COMPARISON.ru.md \
  --output rutech-specialist.cmf
```

Реальный прогон по шагам — Qwen2.5-0.5B-Instruct (q8) на русских доках
этого же репозитория, Apple M4, **8.8 минуты от и до**:

```

  [A] шаг  60: L1=0.020 прунинг= 2% hard-PPL=21.110 (дно 21.110@2%)   <- дно денойзинга
  [A] шаг  90: L1=0.025 прунинг= 6% hard-PPL=22.778 (дно 21.110@2%)
  [A] шаг 120: L1=0.030 прунинг=10% hard-PPL=25.610 (дно 21.110@2%)
  [A] шаг 180: L1=0.040 прунинг=16% hard-PPL=49.659 (дно 21.110@2%)   <- за дном качество рушится
[A] 314s: masked-PPL 21.110                                           <- дно восстановлено из чекпойнта
  [B] шаг  30: held-PPL 18.304
  [B] шаг  60: held-PPL 17.840
  [B] шаг  90: held-PPL 17.474
  [B] шаг 120: held-PPL 17.423                                        <- FCD копает дальше
=== bake: baseline 24.157 | маска 21.110 | маска+FCD 17.423 | прунинг 2% -> СПЕЦИАЛИСТ <= baseline

```

Три вердикта, которые стоит прочесть дважды:

- Тренировочная реплика и реальный движок согласны (baseline 24.157 vs
  24.173) — то, что оптимизирует запекание, и есть то, что отдаёт
  рантайм. Разрыв между 17.4 (f32-реплика) и 19.0 (записанный файл) —
  это q8-переквантование обученного FFN: измерено, не спрятано.
- **Обобщение**: на русском тех-документе, которого НЕ было в корпусе
  (`PERFORMANCE_ROADMAP.ru.md`), специалист даёт 22.56 против 25.62 у
  базы — **−12.0% на невиданном тексте**.
- Дно денойзинга здесь легло на 2% (PPL базы 24 — домен не так уж
  слаб), поэтому выигрыш размера мал (479 → 472 МБ); история — в
  качестве. На слабом домене (PPL базы 70) тот же рецепт режет 11%
  и снимает четверть перплексии — см. следующий раздел.

## Меньше исходной модели — и лучше: запекание DTG-MA

Самая сильная форма навыка не едет рядом с базой — она *заменяет* её
для одного домена. Рецепт DTG-MA (заявка USPTO 19/452,464 —
юридический контекст в [PATENTS.md](../PATENTS.md)) обучает L1-регуляризованную
маску по FFN-нейронам на вашем корпусе, ловит *дно денойзинга* (выброс
шумовых нейронов сначала УЛУЧШАЕТ модель, и лишь потом начинает вредить),
полирует последние слои против точного учителя (FCD), а `--defrag`
запекает автономный файл, где выброшенные нейроны физически отсутствуют —
не хранятся и не вычисляются (claims 9/10 заявки 19/452,464):

```sh
# нативно, одной командой (см. предыдущий раздел):
cortiq skill bake base.cmf --files corpus1.txt corpus2.txt --output specialist.cmf

# или исходный torch-рецепт (те же фазы; полезен на CUDA-машинах):
python3 converter/make_skill_l1fcd.py --model <hf_snapshot_dir>   --id ru --files corpus1.txt corpus2.txt --out skill-ru
cortiq convert --model <hf_snapshot_dir> --defrag skill-ru   --quant q8_2f --output ru-specialist.cmf
```

Измерено насквозь через рантайм этого репозитория на Qwen3.5-0.8B, оценка
на отложенном русском техническом документе, которого рецепт не видел:

| | размер | PPL (ru-тех, held-out) | decode |
|---|---:|---:|---:|
| исходный чекпойнт (bf16) | 1.6 ГБ | — | — |
| CMF q8_2f базовая линия | 733 МБ | 13.97 | 86.0 tok/s |
| **ru-специалист (маска 11% + FCD, defrag)** | **705 МБ** | **11.92 (−14.7%)** | **89.7 tok/s** |

Меньше исходной LLM в 2.3×, лучше на домене и быстрее — из одного
запекания. Собственный отчёт рецепта на своём in-domain held-out ушёл
дальше: дно маски 70.5 → 54.2 PPL при 11% прунинга, −38.7% после FCD,
−19.4% на независимом невиданном тех-корпусе.

Два честных правила, которым научили замеры:

- **Запекание блистает там, где база слаба.** Русский тех-текст (PPL
  базы 70) дал полный эффект; тот же рецепт на домене, где база и
  так хороша (код, PPL 8.7), сделал *хуже*. Сначала проверьте PPL
  базы на домене.
- **Дно денойзинга реально и узко.** Held-out PPL маски по ходу
  обучения: 67.9 при 0% → **54.2 при 11% (дно)** → 77.1 при 18%.
  Рецепт останавливается на дне сам; не жмите дальше ради размера.

`cortiq skill add --sparse <keep>` — быстрое приближение маски *без
обучения* (per-layer top-K по активационной массе задачи) — годится для
разведки, но измерено честно: на калибровке из 8 промптов оно рухнуло
при keep 0.5 (PPL 59 против 11.9). Настоящий метод — обученная маска;
флаг — разведчик.

## Ресурсы: что нужно и на каком железе

**Путь 2 (`skill add`) — лёгкий.** GPU не нужен вовсе:

| Что | Стоимость |
|---|---|
| Скачивание донора | один HF-чекпойнт (~1 ГБ bf16 для 0.5B), кэш в `~/.cache/cortiq/hf` |
| Время | минуты на ноутбуке: квантование привитых тензоров + перезапись файла |
| Память | ~размер базы + один тензор донора (донор читается через mmap) |
| Рост файла | только привитые тензоры: 314 МБ на FFN-навык всех слоёв у q8 0.5B |
| Цена в рантайме | ноль, пока неактивен; активный навык = скорость базы (те же формы, те же ядра) |
| Цена роутинга | один prefill-проход промпта до φ-слоя |

**Путь 1 (`skill bake`) — обучение на CPU.** Запекание держит
f32-реплику модели (4 байта на параметр) плюс оптимизатор только для
полируемых слоёв, поэтому потолок определяет обычная RAM:

| База | RAM запекания (оценка: 4 байта/параметр + оптимизатор) | Время (M4, 240+120 шагов) |
|---|---:|---:|
| 0.5B | ~3 ГБ | 8.8 мин (замерено) |
| 1–2B | ~6–10 ГБ | ~20–40 мин |
| 4B | ~18 ГБ | ~1.5 ч |
| 7–8B | ~35 ГБ | ~3 ч |
| 27B | ~115 ГБ — Mac Studio 128 ГБ+ или torch-рецепт на CUDA | ночь |

Корпус: хватает 100–200 КБ текста домена (реальный прогон выше — 82
чанка по 256 токенов). Для «скилла по бенчмарку» соберите условия задач
и эталонные решения в текстовые файлы и передайте их в `--files`.

## Как уменьшить навык на диске

Навык должен стоить столько, сколько файнтюн реально изменил —
нетронутые нейроны не заслуживают байтов. За это отвечают два флага, оба
измерены и оба проверяются через `--quality`:

**`--min-delta <x>` — не хранить то, чего тюн не касался.** Каждый
кандидатский тензор сравнивается с базой через референс-декодер;
тензоры с относительным изменением ниже порога выбрасываются — рантайм
читает там запись базы, а в доноре лежит ровно то же самое. Гейт
печатает полное распределение дельт — вы видите, прежде чем резать:

```
# донор ru, измерено:
delta gate ≥ 0.0000001: kept 72 / dropped 0 unchanged tensor(s) (−0.0 MB);
rel-delta min 0.0980 / median 0.2196 / max 0.3724
```

Калибруйте порог по *шумовому полу квантования* вашей базы: запеките
самопрививку (`--from` = исходный чекпойнт самой базы) с
`--min-delta 0.0000001` и посмотрите напечатанную медиану — это чистая
ошибка кванта (измеренная медиана 0.92% для нашей q8-базы;
самопрививка с `--min-delta 0.02` корректно выбрасывает все 72 тензора и
отказывается запекать). Дельты этого уровня — шум, а не сигнал. Все три
донора демо лежат выше него всюду (sql 0.9–2.6%, thinker 1.8–3.5%,
ru 9.8–37%), поэтому гейт справедливо сохранил их целиком — он
окупается, когда тюн локален (LoRA, влитая в несколько проекций; тюн,
тронувший только глубокие слои), и никогда не врёт: сначала измеряет.

**`--skill-quant <enc> [--mean-bits N]` — более дешёвая кодировка для
оверлея.** Потензорные dtype нативны для CMF (спека §3), так что навык
может жить в меньшем числе бит, чем база. Измерено на навыке `ru`
поверх q8-базы (отложенная русская проза, PPL базы 11.88):

| кодировка | размер | PPL навыка | вердикт |
|---|---:|---:|---|
| q8 (как база) | 314 МБ | 11.03 (−7.1%) | полное качество |
| `--skill-quant vbit --mean-bits 6` | 257 МБ (−18%) | 11.19 (−5.8%) | сохраняет почти весь выигрыш |
| `--skill-quant vbit` (4.25 бита) | 184 МБ (−41%) | 12.17 (+2.5%) | **хуже базы** |
| `--skill-quant q4` | 176 МБ (−44%) | 12.20 (+2.7%) | **хуже базы** |

Урок тот же, который квантование преподаёт всегда, только уровнем ниже:
*сигнал файнтюна* должен пережить добавленный шум. SFT-дельта 0.5B
(здесь 10–37% относительных) переживает ~6 бит и тонет на 4. После
сжатия всегда перемеряйте `--quality` — число в любом случае ложится в
реестр, и файл несёт собственный вердикт.

Можно и вручную ограничить слои (`--layers 12-23` — вдвое меньше);
правило то же — проверьте, выживает ли нужное вам поведение.

## Паспорт навыка — что положить рядом при передаче

Навык уже сегодня несёт в реестре всё для честного обмена; если вы
публикуете навык (или готовитесь к витрине skill-hub), минимальный
паспорт выглядит так:

| Поле | Откуда | Зачем |
|---|---|---|
| `id`, `name` | `skill add --id/--name` | идентичность в реестре |
| происхождение | репозиторий+ревизия донора или список файлов корпуса запекания | воспроизводимость |
| лицензия донора | карточка HF-чекпойнта | право распространять веса-дельты |
| `quality` | `--quality held-out.txt` — пишется в реестр | измеренный вердикт: база → с навыком, на чём |
| порог `--min-delta` | команда запекания | сколько тензоров отброшено как шум кванта |
| кодировка | `--skill-quant`/`--mean-bits` | читатель видит, пережил ли сигнал сжатие |
| промпты роутинга | `--prompts file.txt` | φ-дескриптор: как файл узнаёт «своих» |

Всё, кроме лицензии и происхождения, файл несёт сам (`skill list`
показывает); лицензию и источник укажите в описании рядом. Перенос
навыка между файлами одной базы — операция над каталогом, веса не
пересчитываются.

## Как это устроено внутри

- Тензоры навыка — обычные записи каталога с именами `skill.{id}.{tensor}`:
  mmap как всё остальное, подгрузка страниц по первому касанию. Хранение
  растёт как |база| + Σ|дельт| ([спека §9](CMF_V2_SPEC.ru.md)).
- Рантайм разрешает каждый тензор через индирекцию источника: запись
  базы или `skill.{active}.{name}` — замещение, никогда не сложение;
  полная модель навыка никогда не собирается.
- Роутинг — recon-argmin по аффинным подпространствам: без обученного
  гейта и классификатора — выбор является *свойством файла*. `route`,
  `explain`, `--blend`, `--route-dynamic` читают одни и те же дескрипторы
  по ~4 КБ.
- Поле `quality` реестра — честный контракт (claim 16,
  [PATENTS.md](../PATENTS.md)): что измерено, на чём и с каким
  результатом. `skill list` его показывает.

В CLI слово `backbone` осталось как значение флага (`--skill backbone` =
принудительно базовая модель) и как ключ в JSON-отчётах — это
программные идентификаторы, они стабильны.
