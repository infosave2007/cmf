# CMF — Cortiq Model Format

**Однофайловый формат LLM, у которого память внимания перестаёт расти вместе с контекстом.**

[![CI](https://github.com/infosave2007/cmf/actions/workflows/ci.yml/badge.svg)](https://github.com/infosave2007/cmf/actions/workflows/ci.yml)
[![crates.io](https://img.shields.io/crates/v/cortiq-core.svg)](https://crates.io/crates/cortiq-core)
[![downloads](https://img.shields.io/crates/d/cortiq-cli.svg)](https://crates.io/crates/cortiq-cli)
[![docs.rs](https://img.shields.io/docsrs/cortiq-core)](https://docs.rs/cortiq-core)
[![License: Apache-2.0](https://img.shields.io/badge/license-Apache--2.0-blue.svg)](https://github.com/infosave2007/cmf/blob/master/LICENSE)

Файл `.cmf` несёт в себе веса, токенизатор и шаблон чата разом, сам проверяет
свою целостность и отображается в память прямо с диска. Движок — небольшое ядро
на Rust, под которым нет никакого ML-фреймворка: ни torch, ни BLAS, ни ONNX, ни
установленной CUDA, ни компилятора C++. На CPU он работает везде, на GPU — через
wgpu (Vulkan / DX12 / Metal) при сборке из исходников. Конвертация модели — одна
команда и ни строчки Python.

Чем он отличается: **одним флагом внимание модели превращается в потоковый
оператор с постоянной памятью** — без дообучения, веса побайтово те же, — и
длинный диалог перестаёт стоить больше памяти, чем короткий.

## Попробуйте

```sh
# готовая программа: github.com/infosave2007/cmf/releases/latest
# или, если установлен Rust:
cargo install cortiq-cli

cortiq convert --model Qwen/Qwen3-0.6B --quant q8 --output qwen.cmf
cortiq run qwen.cmf --prompt "What is the capital of France?" --greedy --no-think
```

```console
Loading model: qwen.cmf
Ready: qwen3 | Task: general | Sparsity: 0%

Prompt: What is the capital of France?

The capital of France is **Paris**.
[10 tokens, 40.1 tok/s, finish: stop]
```

`convert` скачивает модель с Hugging Face (её куски — параллельно), квантует её и
пишет один самодостаточный файл — нативный Rust, без torch, без numpy. Уже есть
GGUF? `cortiq import-gguf <file-or-repo-id> --output model.cmf` читает и его,
тоже нативно.

`run` применяет шаблон чата, хранящийся в файле, так что это настоящий ход
диалога, и модель останавливается сама. Qwen3 — рассуждающая модель: уберите
`--no-think`, и она сперва покажет своё рассуждение в `<think>`. `--raw`
полностью пропускает шаблон (режим дополнения). `Task` и `Sparsity` показывают
наложенный навык; если навык не выбран, там `general` / `0%` — подробнее о
[навыках](#много-специалистов-одна-базовая-модель) ниже.

**Запустится ли ваша модель?** Нативная конвертация на сегодня: qwen2 · qwen3 ·
qwen3.5 (включая слитую раскладку qwen3_next / AgentWorld) · llama · mistral ·
qwen-moe — dense, MoE и GatedDeltaNet. Пока нет: gemma, phi, deepseek. Всё
остальное — пробуйте `import-gguf`; и если он откажется, это ошибка, о которой
стоит завести задачу.

## Встройте в то, чем уже пользуетесь

`cortiq serve` говорит по OpenAI API, так что существующие клиенты и SDK работают
без изменений — просто направьте их на него:

```sh
cortiq serve qwen.cmf --port 8080        # + веб-панель на /
```

```sh
curl localhost:8080/v1/chat/completions -H 'Content-Type: application/json' -d '{
  "model": "cmf",
  "messages": [{"role": "user", "content": "Explain mmap in one sentence."}]
}'
```

`/v1/models`, `/v1/completions` и `/healthz` тоже на месте, потоковая выдача
(`"stream": true`) работает. Поле `model` обязательно по схеме, но ни с чем не
сверяется — присылайте то, что присылает ваш клиент.

Трезво оцените область применения, прежде чем разворачивать: **запросы
обрабатываются последовательно** (по одному на модель) и **аутентификации нет** —
это сервер для локального использования, а не многопользовательский шлюз. Не
выставляйте его в сеть, которой не доверяете.

## Почему CMF

### Внимание, которое перестаёт расти вместе с контекстом

Обычно каждый добавленный в диалог токен навсегда оседает в KV-кеше. `--o1`
заменяет softmax-внимание слоя на потоковый оператор, который вместо этого держит
**состояние фиксированного размера**: несколько первых токенов хранятся точно
(якоря), последние несколько сотен — тоже точно (окно), а всё, что старше, живёт
в сжатой выжимке по опорным точкам, — и всё это под одним общим знаменателем
softmax. Конвертация мгновенна, и **веса не меняются вовсе** — флаг лишь
записывает подсказку в заголовок.

Замерено на **Qwen3.5-4B** (24 слоя GatedDeltaNet + 8 softmax-слоёв; `--o1 all`
конвертирует эти 8; 16 query-голов / 4 KV-головы, head_dim 256; q8_2f). Apple M4,
машине давали остыть между прогонами:

| контекст | память внимания, `--o1 off` | `--o1 all` | декодирование, `off` → `all` |
|---:|---:|---:|---:|
| 543 | 141.0 MB | **124.1 MB** | 15.7 → 16.5 tok/s |
| 1055 | 174.5 MB | **124.1 MB** | 15.5 → 16.5 tok/s |
| 4127 | 380.3 MB | **124.1 MB** — в 3.1× меньше | 8.2 → 10.7 tok/s |

**124.1 MB на любой длине контекста** — в этом весь смысл. Раскладывается это на
постоянный «пол» рекуррентных слоёв плюс фиксированные **18.8 MB**, замещающие
KV-кеш softmax-слоёв. Иначе этот KV рос бы со скоростью ~64 KiB/token, так что две
кривые пересекаются примерно на **290 токенах**: ниже этой точки `--o1` обходится
вам в несколько лишних мегабайт, выше — только экономит: 3.1× меньше на 4k и ~17×
на 32k по экстраполяции (состояние постоянно, поэтому отношение продолжает расти;
мы мерили до 4k — запустите `cortiq bench model.cmf --ctx 32768` на своей машине).

**Чего это стоит.** Выжимка — это приближение, и платить за неё приходится
качеством: перплексия растёт в **1.13×** на Qwen3.5-4B и в **1.30×** на
Qwen3-0.6B (сконвертированы все 28/28 слоёв) — замерено на отложенном wikitext,
через настоящее потоковое ядро, на самом невыгодном для него участке: опорные
точки строятся по первым 256 токенам и дальше не обновляются, а оцениваются
только позиции, успевшие уйти за пределы точного окна, — то есть ровно те, где
работает одна лишь выжимка. Чем больше в модели softmax-внимания, тем дороже
обходится `--o1`: у гибрида есть рекуррентные слои, которые несут дальний
контекст, а в модели с чистым вниманием всю работу приходится делать выжимке.
Относитесь к `--o1` как к регулятору «память/качество», а не как к бесплатному
выигрышу. С ростом контекста цена не растёт — состояние ведь тоже не растёт. Не
верьте нам на слово ни в чём из этого; померьте на своей модели:

```sh
cortiq ppl model.cmf --file wiki.txt --o1 all
```

Он прогоняет сконвертированную модель через настоящее потоковое ядро и рядом
печатает базовую линию точного внимания на тех же самых токенах, так что
отношение — это сопоставимое измерение, а не заявление.

Если для вашего сценария такая цена слишком высока, `cortiq fcd` отыгрывает часть
её ограниченным нативным проходом обучения — см. [O(1) в деталях](#o1-в-деталях).
Чистой цифры «до/после» для него мы пока не публиковали.

Чтобы не было недопонимания насчёт оси, по которой идёт речь: `llama.cpp` — это
эталон, с которым мы себя сверяем, и с версии 0.3.0 CPU-движок держится и на
коротком одиночном потоке. Один полный сопоставимый прогон (16 июля 2026:
Qwen2.5-0.5B-Instruct, Apple Silicon, только CPU, по 8 потоков у обоих, точное
внимание у обоих, `llama.cpp` b9310 против CMF 0.3.0, по 3+ серии отдельными
процессами):

| CPU, 8 потоков | `llama.cpp` (q8_0) | CMF (q8) | Δ |
|---|---|---|---|
| Обработка промпта (pp512) | 220.6 ± 2.5 tok/s | 374–390 tok/s | **+70%** |
| Генерация (tg128) | 60.4 ± 1.1 tok/s | 95–101 tok/s | **+60%** |
| Качество кванта (PPL к своему f16, 12×512 окон) | почти без потерь | +0.38% | выровнено |
| Размер файла | 644 МБ | 479 МБ | **−26%** |

Одна модель на одной машине — это точка данных, а не лидерборд: квантования
одного класса, но не побайтно одинаковые (8.06 против 8.5 бит/вес), и полная
матрица (модели × размеры × процессоры, bootstrap-CI) ещё впереди. Методика
едет вместе с репозиторием — `cortiq bench --json` печатает tok/s вместе со
счётчиками аллокаций и диспетчей планировщика на токен — так что проверьте на
своём железе, а не верьте на слово. А за пределами короткого случая CMF делает
плоским рост расходов с длиной контекста: точное внимание проседает с 15.7 до
8.2 tok/s по мере роста контекста, тогда как `--o1` держится на ~16.5. Таблица —
про короткий случай, `--o1` — про хвост длинного контекста.

### Один файл, ничего рядом

Токенизатор (побайтовый BPE от Hugging Face) и шаблон чата (Jinja) путешествуют
**внутри** модели — GGUF делает так же, и правильно делает: поведение в чате
задаёт файл, а не ваша программа, и рядом нет отдельных файлов-спутников,
которые можно потерять или незаметно рассогласовать по версиям. Сверх этого
`.cmf` добавляет целостность: фиксированный 128-байтовый заголовок — «конверт»,
в котором записано, где что лежит, — плюс 64-битный хеш на каждый тензор
означают, что `.cmf` либо валиден, либо `open()` громко падает с ошибкой. Это
ловит обрезанный файл и битовую порчу; это не подпись.

```sh
cortiq verify model.cmf     # конверт, секции, хеш каждого тензора
cortiq info   model.cmf     # архитектура, тензоры, квантование, навыки
```

Веса отображаются в память и читаются на месте, поэтому старт мгновенный, а
неиспользуемые веса вообще не попадают в RAM. Квантование задаётся потензорно и
смешивается — `q8` (1 byte/param) · `q8_2f` (int8 с масштабом и по строкам, и по
столбцам — лучше качество при том же числе байт) · `q4` (0.5) · `f16` · `vbit`
(переменная 3–8 бит, ~4.25 в среднем ≈ 0.53), — так что в одном файле можно
оставить внимание на q8, а FFN опустить до q4.

### Много специалистов, одна базовая модель

Выкатывать *N* дообученных версий обычно означает *N* полных копий на диске и в
оперативной памяти. CMF держит **одну базовую модель плюс по одному небольшому
навыку на специалиста**: навык хранит только те тензоры, которые он реально
замещает, и при работе движок читает их *вместо* тензоров базовой модели —
отдельная модель никогда не собирается. Объём хранения — это
`|базовая модель| + Σ|навыки|`, а не `N × |модель|`, а навык, которым вы не
пользуетесь, занимает **ноль оперативной памяти**.

Навык не просто дешевле выкатывать — на своей задаче он обыгрывает ту самую
базовую модель, поверх которой лежит: на отложенных данных наложенный навык
снижает перплексию на задаче на **24.9%**
([спецификация §9](docs/CMF_V2_SPEC.md)). Навыки окупаются сильнее всего там, где
базовая модель слаба; на темах, с которыми она и так справляется, ждите меньшего.

```sh
cortiq run model.cmf --prompt "SELECT ..." --skill sql
```

Не хотите выбирать вручную? `cortiq route` подберёт навык по запросу, а
`cortiq explain` покажет, почему именно его.

Обслуживание *N* специалистов под задачи:

| | N полных дообучений | база + N внешних LoRA | **CMF** |
|---|---|---|---|
| На диске | N × полная модель | база + N адаптеров (отдельные файлы) | одна базовая модель + N небольших навыков, **один файл** |
| Токенизатор + шаблон чата | на каждую копию / отдельным файлом | встроены, если база в GGUF, иначе отдельным файлом | **встроены** |
| Потензорный хеш целостности | — | — | **да** |
| Неиспользуемый навык в RAM | загружен | 0 с сервером, подкачивающим адаптеры (S-LoRA / vLLM); иначе загружен | **0**, подкачка по мере использования, серверный стек не нужен |
| Навык едет внутри файла модели | — | нет (отдельные файлы адаптеров) | **да, под той же цепочкой хешей** |

Полное сравнение формат за форматом — GGUF, safetensors, ONNX, PyTorch, GGML,
TensorRT, с явно расписанными компромиссами — в
[docs/COMPARISON.md](docs/COMPARISON.md).

## Установка

```sh
cargo install cortiq-cli                 # утилита командной строки `cortiq`
cargo add cortiq-core                    # или работа с форматом из своего кода на Rust
```

Готовые программы — в [последнем выпуске](https://github.com/infosave2007/cmf/releases/latest):
Linux x86-64, macOS (Apple Silicon и Intel), Windows (x86-64 и ARM64); к каждому
архиву прилагается `.sha256`. Они **только под CPU**; для работы на GPU собирайте
из исходников с `--features gpu`.

## Команды

| команда | что делает |
|---|---|
| `cortiq convert --model <hf-repo\|dir>` | модель с Hugging Face → `.cmf` (нативно на Rust) |
| `cortiq import-gguf <file\|hf-repo>` | GGUF → `.cmf`, все распространённые ggml-кванты |
| `cortiq run model.cmf` | чат, или `--prompt` для одиночного вопроса |
| `cortiq serve model.cmf` | HTTP-сервер, совместимый с OpenAI, + веб-панель |
| `cortiq info` · `masks` · `verify` | осмотр архитектуры, тензоров, навыков; проверка целостности |
| `cortiq bench --ctx 4096` | tok/s и память на заданном контексте |
| `cortiq ppl --file f.txt` | перплексия по готовому тексту — ворота качества |
| `cortiq fcd` | восстановительный тренер для `--o1`-моделей (якорь по KL, ворота по генерации) |
| `cortiq diff a.cmf b.cmf` | что изменилось между двумя версиями модели |
| `cortiq route` · `explain` | какой навык выбирает маршрутизатор и почему |

`cortiq <command> --help` документирует каждый флаг.

### Конвертация

```sh
cortiq convert --model Qwen/Qwen2.5-0.5B-Instruct --quant q8    --output model.cmf
cortiq convert --model ./my-hf-checkpoint         --quant q8_2f --output model.cmf
cortiq import-gguf Qwen/Qwen2.5-0.5B-Instruct-GGUF --output model.cmf --quant q8
```

Импорт GGUF покрывает `Q4_0/1`, `Q5_0/1`, `Q8_0`, `Q2_K`…`Q6_K`, `IQ4_NL/XS` и
`BF16`.

Нативный конвертер пишет **базовые модели**. Python-инструментарий в `converter/`
по-прежнему остаётся тем, что производит описанные выше тензоры-замены под каждый
навык и маски задач, а также GPTQ-калиброванный вариант v-bit, которому нужен
гессиан активаций. Путь v-bit только по весам — нативный.

## O(1) в деталях

Запишите подсказку на этапе конвертации или решайте на этапе загрузки — движок
подхватывает подсказку из заголовка автоматически:

```sh
# на конвертации: все softmax-слои, N самых глубоких или явный список
cortiq convert --model Qwen/Qwen3-0.6B --quant q8 --o1 all    --output model.cmf
cortiq convert --model Qwen/Qwen3-0.6B --quant q8 --o1 deep12 --output model.cmf

# или переопределить при загрузке, без переконвертации
cortiq run   model.cmf --o1 all      # принудительно преобразовать все softmax-слои
cortiq run   model.cmf --o1 off      # вернуться к точному вниманию
cortiq bench model.cmf --ctx 4096    # память и tok/s, с флагом и без
CMF_O1=deep6 cortiq serve model.cmf  # переопределение через env, тот же синтаксис

# тонкая настройка (проверенные умолчания: 32 опоры, окно 128, 4 якорных ключа)
cortiq run model.cmf --o1 all --o1-m 32 --o1-window 128 --o1-sink 4
```

На гибридных моделях (например, qwen3.5: слои GatedDeltaNet с островками softmax)
`--o1 all` конвертирует только softmax-слои, и это делает состояние внимания всей
модели постоянным по длине контекста.

**Восстановление.** `cortiq fcd` — это ограниченный нативный проход обучения (без
Python, без ML-фреймворка), который настраивает только norm/FFN-тензоры
сконвертированных слоёв, ориентируясь на ту же модель с точным вниманием (якорь
по KL), и сохраняет промежуточный результат только в том случае, если генерация
на длинном контексте не сваливается в петли:

```sh
cortiq fcd model.cmf --corpus corpus.txt --gen-check --gen-gate --out model.fcd.cmf
# ручки: --steps 300 --eval-every 25 --kl 0.7 --lr 5e-5 --o1 all|deepN|i,j,k
#        --val-corpus val.txt --gate-threshold 0.35 --gate-slack 0.10
```

## Формат

`.cmf` — это фиксированный 128-байтовый конверт, за которым идут секции; читатель
адресует их **только** через этот конверт и никогда не полагается на их порядок:

- **header JSON** — архитектура, умолчания квантования, комплект чата, реестр навыков, происхождение
- **каталог тензоров** — бинарные записи по 56 байт (name, dtype, shape, offset, nbytes, hash64), читаемые без обращения к JSON
- **блоб весов** — выровнен по страницам, отображается и читается на месте
- **навыки** — битово упакованные маски задач и тензоры-замены под каждый навык
- **токенизатор** — дословный файл Hugging Face
- **разреженный индекс** — предвычисленный

Также поддерживаются: головы многотокенного предсказания (MTP), FFN-слои со
смесью экспертов (MoE), рост навыков только добавлением (append-only) с
уплотнением, и разрезание модели на `N` файлов, каждый из которых валиден сам по
себе.

**Вы ничем не связаны.** `python/cmf_reader.py` — это полноценный читатель на
~300 строках стандартной библиотеки и numpy, не разделяющий ни строчки кода с
движком на Rust: он намеренно написан по спецификации, чтобы доказать, что
формат переживёт эту реализацию:

```python
from cmf_reader import CmfReader
r = CmfReader("model.cmf")
w = r.tensor("model.layers.0.mlp.gate_proj.weight")   # np.ndarray, деквантованный
assert r.verify() == []                               # все хеши тензоров сходятся
```

Если завтра этот проект исчезнет, ваши веса всё равно можно прочитать по одной
лишь спецификации. Полная нормативная спецификация — в
[docs/CMF_V2_SPEC.md](docs/CMF_V2_SPEC.md).

## Откуда это взялось

Идеи, лежащие в основе конструкции, выросли из отдельной работы автора над
физической теорией — вакуумной массовой фракции (VMF) в рамке гравитации нулевого
вектора (NVG): модель «общая основа плюс небольшие возмущения», двухполевое
квантование. Ничто в формате не зависит от того, верна ли эта теория; формат
стоит на спецификации и на цифрах выше. Отображение, с жёсткой границей между
тем, что *измерено*, и тем, что остаётся метафорой:
[принципы VMF/NVG в основе CMF](VMF_principles_in_CMF.ru.md)
([English](VMF_principles_in_CMF.md) · [中文](VMF_principles_in_CMF.zh.md)).
Сама физика живёт в [отдельном репозитории](https://github.com/infosave2007/vmf).
